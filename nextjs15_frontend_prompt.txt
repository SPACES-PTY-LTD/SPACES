You are a senior frontend engineer. Build a new Next.js 15 (App Router) frontend that connects to the API endpoints listed below. Use Tailwind CSS and shadcn/ui components throughout. The public landing page design must be clearly inspired by the Nexus Work Management Platform template: https://v0.app/templates/nexus-work-management-platform-ALfQrxyrJ8b . The admin area UI must follow the shadcn dashboard example: https://ui.shadcn.com/examples/dashboard .

Deliverables
- A Next.js 15 project scaffold with App Router and TypeScript.
- Tailwind + shadcn/ui configured and used across all pages.
- Landing page (public) inspired by the Nexus template (layout, sections, and visual rhythm) but original copy.
- Auth flows and admin area with role-based access: Admin and Super Admin.
- A complete set of admin pages aligned to the API endpoints below, with sidebar navigation and routes.
- API client layer with auth handling, typed responses, and loading/error states.

Project Setup Requirements
- Use `app/` router, server components by default, client components only where needed.
- Configure Tailwind and shadcn/ui (button, card, table, dialog, sheet, badge, tabs, dropdown, toast, input, select, textarea, date picker where appropriate).
- Use `@/` alias for imports.
- Add `.env.local` with `NEXT_PUBLIC_API_BASE_URL` and `NEXT_PUBLIC_APP_ENV`.
- Create a thin API client using fetch with token handling (access + refresh) and a safe retry flow.
- Persist auth tokens in httpOnly cookies via Next.js route handlers (preferred), or fallback to secure storage.
- Protect routes by role and redirect unauthenticated users.

User Roles
- `admin` and `super_admin` roles.
- Both use the same login form, but redirect to different landing dashboards.
- Super Admin has access to all admin pages; Admin sees only non-super-admin sections.

Notes

Next.js 15 Guidance (avoid legacy patterns)
- Use App Router only. Do not create a `pages/` directory.
- Prefer Server Components by default; only add `"use client"` where needed.
- Use Route Handlers (`app/api/*/route.ts`) for server-side auth cookie management and token refresh flows.
- Avoid global middleware for auth gating unless absolutely necessary; prefer layout-level checks and server actions.
- Use Server Actions for mutations where appropriate; don’t rely on `getServerSideProps` or `getInitialProps`.
